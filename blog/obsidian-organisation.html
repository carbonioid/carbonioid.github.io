notes on obsidian organisation

Some eye candy: my current obsidian graph. Nodes are notes, and edges are the links between them.

For those who don't know, Obsidian is a markdown editor which lets you organise notes by links, folders and tags. For me it has become a space where I put (nearly) everything I do which can be written down textually: school notes, disorganised thoughts, independent research on my interests, you name it. Even this blog post was first written in my Obsidian vault.

The main advantage of having one central store for these notes (as Obsidian calls them) is that it makes connecting ideas and data together very easy. Obsidian also has a massive plugin ecosystem, which allows you to add functionality to your client. For example, each day a plugin creates a note which collates my to-do list for the day, any large assignments I'm currently doing, and a list of media I'm currently reading/watching. That's only possible because the data is all in one place and, crucially, it's structured well.

The main focus of this post will be how I structure my Obsidian vault. Markdown is a useful format because it allows both human- and machine-readable properties, so I'll discuss both the more holistic approaches I've taken to human-readable structure, and the machine-readable properties I try to maintain to make automation in my vault easy.

Tags

A snippet of some of the tags I currently use for organisation.

Your mileage will vary, but between links, folders and tags what I use the most currently is tags. In the frontmatter of each note, there will typically be:

  • one class tag (those are the ones you see in the image above)
  • one (optional) type tag
  • zero or more content tags

Using tags lets me add a lot of structure to my notes (as opposed to links which are more fluid) without constraining the amount of information I can store (as opposed to folders - a note can only belong to one folder at a time.)

The main benefits of having layers of organisational structure are twofold. For class and type tags, they allow me to define how these notes interact with other parts of my system. For example, notes of specific types relating to projects I'm doing are the only ones allowed to have a "progress" tag denoting how far through them I am.

Secondly, the requirement to specify these tags whenever I start a note helps forces me to be intentional when deciding about the structure of the note: should this really be only one note, or does the inability to decide on a proper type imply that I should actually split it into two separate notes to be consistent?

Class tags

Class tags are the highest level of organisation for my notes. A class note dictates the very broad organisational principles a note should follow: what kind of information does the note contain, how long is it, how is it written etc. I use five different classes, and I'll go through each briefly:

  • Knowledge: the knowledge type is the most typical kind of Obsidian note, similar to a literature note in a Zettelkasten. Knowledge notes are atomic and stored learned information. School notes fit into this category.
  • Thought: this is the most particular type I use, as it doesn't fit elsewhere. Thought notes are just notes for storing disorganised thoughts about something. This is usually in a serial format: for example, I might have a thought note about selective abstraction with thoughts about it from various days.
  • Organisation (org): organisation notes which collate information and perform boilerplate functions. They don't contain original information, rather just move around information from other notes. For example, a list of all your notes about Physics would be an org note.
  • Info: info notes are notes which contain information that could just as easily not be on Obsidian. Unlike knowledge notes, they aren't usually atomic and often contain long paragraphs of text. Generally, info notes are checklists and long pieces of writing (this blog post was originally an info note.)
  • Working note: this is my most recent category and is somewhat similar to a fleeting note in a Zettelkasten. Working notes are messy and in progress, usually a store of information for a project I'm working on. A working note eventually gets split into one or more notes and reorganised into another kind of note (usually a knowledge note.) This category is really here out of necessity rather than any organisational reason, just because I tend to have a lot of these working notes.

Okay, sorry for that complete text wall. The next sections should be shorter and easier to explain: type tags and content tags

Type tags

Type tags indicate the structure of a note. That is, types of notes with distinct structures all have a common type tag. Needing to set the type whenever I start a note helps me be a lot more thorough and structured in my note-taking. Type tags are also very useful when creating notes from templates: you can enter a type tag in the template, then be sure that all tags created with that template will have a common tag.

To give some examples, I keep notes for media I'm reading/watching. All of these get the #meta/type/media-note tag, as the structure of the frontmatter is consistent between all media notes (although not necessarily the text of the note.) This could also go the other way - the frontmatter might vary, but the text's structure would be the same.

Content tags

Content tags dictate, well, the content of a note. They are mostly used when I'm searching for notes, and take the place of what other people would probably use folders for: there is a maths content tag for all my mathematics-based notes, as well as an obsidian tag for all my notes talking about obsidian.

Links

Links aren't as important to me as they perhaps should be. In my daily notes, I make a lot of use of linking to notes for people and projects I'm working on (although I haven't found a useful way to use this data in queries yet.) I'm also making more of a start making mathematics theorems and proofs atomics so I can link between them. Overall, links more or less just function as links for me: they let me reference other pages.

Folders

In terms of folders, when I started out with Obsidian I used many, many nested folders to organise all my notes. I've since moved away from this system (although I'm still partially migrated from it) mostly due to the fact that a note can only belong to one folder. There is some utility to folders as they provide a hierarchical structure which, crucially, I can view in Obsidian's file browser, but my current hope is to instead just use router pages (see below) and tags.

Router pages

The most useful pattern for Obsidian I'm using by far is router pages. A router page is an organisational page which only serves to list all the pages relating to that topic. The benefit of them is that they just make navigating your vault so much easier: if you don't know the precise name of a file but know the topic, you can just use the quick switcher to get to the router for that topic and quickly find the page.

This more or less allows collections of notes (through folders or tags) to interact as notes in your vault rather than being navigated through the file browser. Normally, I use a dataview query to list notes in a router. Obviously, you shouldn't make a router for topics that are too broad: in my experience, routers get cumbersome after you have more than 10 or 15 notes in them.

The final benefit is that routers allow your notes to act like they're in folders without actually being in folders. As I'm switching over to tags, I don't have to change my navigation habits (in fact, routers actually become more flexible.) I only need to update my dataview queries to query by tag and not enclosing folder.

Frontmatter

Frontmatter is a set of machine- and human-readable key-value pairs at the start of an Obsidian note. Its purpose is to store small, structured pieces of information about the current note. I use frontmatter for three pretty typical uses cases:

Note type-specific properties: Some note types (from the "Type tags" section) will have a set of frontmatter properties that are unique to them: for example, all media notes in my vault have an "author" frontmatter property.

General properties: I use some general properties which all notes can have: namely a custom "related" property which I use to denote related notes, although this differs slightly depending on the context. For example, in notes for math theorems, related notes are often theorems needed as prerequisites.

I also have a "progress" tag which is used over multiple note types (mostly relating to ongoing projects). This is very useful for organising notes in dataview queries.

As a quick addendum, I would recommend anyone who uses a lot of frontmatter tags and has an interest in making their vault pretty to install Typify to make them look nice.
